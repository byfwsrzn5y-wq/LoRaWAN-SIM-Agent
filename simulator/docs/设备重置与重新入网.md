# 设备重置与重新入网

说明 OTAA/ABP 重置语义及 HTTP 控制接口（`POST /reset`）。OTAA/ABP 设备支持运行中重置，无需重启模拟器。

## 一、OTAA 重置（重新入网）

重置后，该节点**清除当前会话**（DevAddr、NwkSKey、AppSKey），**下一包上行**会重新发送 **Join Request**，收到 ChirpStack 的 Join Accept 后再次派生会话并开始发 Data。

适用场景：修改 ChirpStack 侧设备配置（如 AppKey、Profile）后，让模拟器端“重新入网”以使用新配置。

| 行为 | 说明 |
|------|------|
| `joined` | 置为 `false` |
| `devAddr` / `nwkSKey` / `appSKey` / `devAddrHex` | 删除，并从 `globalDeviceMap` 移除旧 DevAddr |
| `devNonce` | 默认置 0（可选项保留不重置） |
| 保留 | `appEui`、`devEui`、`appKeyBuf`、`isOtaa`、`macParams`、`nodeState` 等 |

## 二、ABP 重置

仅重置**上行帧计数** `fCntUp` 为 0，并同步到 FCnt 持久化（若启用）。不修改 DevAddr/密钥。

## 三、如何触发重置

### 1. HTTP 控制接口（推荐）

在配置中开启控制服务：

```json
"controlServer": {
  "enabled": true,
  "port": 9999
}
```

- **重置单个设备**（按 DevEUI）  
  - `GET /reset?devEui=0102030405060701`  
  - 或 `POST /reset`，Body: `{ "devEui": "0102030405060701" }`  
- **重置所有 OTAA 设备**  
  - `GET /reset` 或 `POST /reset`，不传 `devEui`  
- **可选参数**（仅 POST Body）  
  - `resetDevNonce`: 是否将 OTAA 的 devNonce 置 0（默认 `true`）  
  - `abpResetFcnt`: 是否重置 ABP 的 FCnt（默认 `true`）

示例（修改配置后重新入网）：

```bash
# 重置指定 OTAA 设备，下一包会发 Join Request
curl -X POST http://127.0.0.1:9999/reset -H "Content-Type: application/json" -d '{"devEui":"0102030405060701"}'

# 重置所有 OTAA 设备
curl http://127.0.0.1:9999/reset
```

### 2. 重启进程

重启模拟器后，所有 OTAA 节点会以 `joined: false` 重新启动，相当于全部重新入网。适合“改完配置后整体重跑”的场景。

## 四、流程小结

1. OTAA 设备已入网并正常发 Data。
2. 在 ChirpStack 中修改该设备配置（如 AppKey、Profile、MAC 等）。
3. 在模拟器侧调用 **重置**（如 `POST /reset` 带 `devEui`，或重置所有 OTAA）。
4. 下一包上行该节点会发 **Join Request**，ChirpStack 下发 **Join Accept**，模拟器派生新会话并继续发 Data。

ABP 设备同样支持重置：通过同一 API 指定其 DevEUI（若控制端能按 DevEUI 寻址），将只做 FCnt 清零，不改变密钥与 DevAddr。
