# 配置与联调总览

本文是 **单一入口说明**：配置文件如何被加载、v2.0 与 legacy 的对应关系、与 ChirpStack Gateway Bridge（UDP/MQTT）对齐的要点、以及 `sim-state` 调试输出。细节核对仍用 [ChirpStack 联调配置核对清单](ChirpStack联调配置核对清单.md) 逐项勾选。

---

## 1. 推荐运行入口

| 用途 | 命令 |
|------|------|
| 生产/联调（MQTT/UDP） | `cd simulator && node index.js -c <配置.json>` |
| 一键脚本：启动模拟器（无前端 UI） | `cd simulator && ./start.sh <配置.json>` |

**`main.js` 已弃用**（仅转发 `index.js`）。运动/环境/衍生异常请在配置中启用并由 `index.js` 加载，见 [`src/runtime/motion-environment.js`](../src/runtime/motion-environment.js) 与 [simulator/README.md 运行入口表](../README.md)。

---

## 2. 配置文件路径解析

模拟器对 `-c` / `--config` 的相对路径按顺序尝试：

1. `process.cwd()` 下路径  
2. `simulator/`（`index.js` 所在目录）下路径  
3. **仓库根**（`simulator` 的上一级）下路径  

因此可在 `simulator` 目录执行：

```bash
node index.js -c configs/50-device-test-config.json
```

若该文件实际在仓库根 `configs/`，也会被正确解析。启动日志会打印 **`[Config] Loaded: <绝对路径>`**。

环境变量：

| 变量 | 作用 |
|------|------|
| `LORAWAN_SIM_LNS_HOST` | 覆盖 v2 `simulation.gateway.address` 解析后的 `lnsHost`（见 `index.js` 中 `normalizeV20ConfigForLegacyIndex`） |
| `LORAWAN_SIM_CONFIG` | 供 `dev-sim-entry.sh` 启动脚本读取，作为 `-c` 的配置路径（CLI 方案不依赖该变量） |

完整示例见根目录 [`.env.example`](../../.env.example)。

## 3. 配置版本 2.0（`version: "2.0"`）

### 3.1 `simulation.gateway`

| JSON 字段 | 映射到 legacy / 运行时 |
|-----------|-------------------------|
| `gatewayEui` | `gatewayEui` |
| `address` | `lnsHost`（可被 `LORAWAN_SIM_LNS_HOST` 覆盖） |
| `port` | `lnsPort`（如 UDP Semtech 常见 **1702**，以你 Bridge 为准） |
| `location`（可选 `{x,y,z}`） | `signalModel.gatewayPosition`（仅当未单独配置 `signalModel` 时合并） |

### 3.2 `devices[]`（扁平 OTAA）

每条设备在规范化后形如 legacy 的 `devices[]` 项：

- `devEui`、`appKey`、`joinEui` → `lorawan.devEui` / `appKey` / `appEui`
- `mode: "otaa"`（默认）  
- `location: { x, y }` → 内部 `position`（**与拓扑世界坐标一致**）
- `interval`（秒）→ `uplink.intervalMs`

实现代码：[`index.js` 内 `normalizeV20ConfigForLegacyIndex`](../index.js)。

### 3.3 MQTT：v2 默认不写 `mqtt` 时

规范化后为 **`mqtt.enabled: false`**，模拟器走 **UDP** 报文转发到 `lnsHost:lnsPort`。

若 ChirpStack 侧使用 **MQTT Gateway Bridge**，必须在同一 JSON 中增加 **`mqtt` 块**（与 legacy 相同），例如：

```json
"mqtt": {
  "enabled": true,
  "server": "tcp://127.0.0.1:1883",
  "mqttTopicPrefix": "as923",
  "marshaler": "protobuf",
  "username": "",
  "password": "",
  "clientId": "lorawan-gw-sim",
  "clean": true,
  "qos": 0
}
```

**必须与 Bridge / `region_*.toml` 中 `topic_prefix` 及 `marshaler` 一致**，否则会出现「模拟器在跑，NS 无上行」。对照表见 [ChirpStack 联调配置核对清单 §2.3](ChirpStack联调配置核对清单.md)。

**可复制示例**：[`configs/examples/chirpstack-mqtt-v2-10nodes.json`](../configs/examples/chirpstack-mqtt-v2-10nodes.json)（10 台 OTAA + MQTT）。

## 4. 展示坐标 vs 射频模型

- **拓扑图**使用的节点/网关坐标来自配置中的 `position` / v2 `location`，以及可选的 `signalModel.gatewayPosition`。  
- 若未配置 `signalModel.gatewayPosition`，模拟器可能根据**节点包络**推断**仅用于展示**的网关位置（`computeVizGatewayPositionFromNodes`），**不等于** `signal_model.js` 里用于 RSSI 仿真的同一套假设。  
- 需要与传播模型严格一致时，请显式配置 `signalModel`。

## 5. `sim-state.json` 调试输出

- 文件路径：`simulator/sim-state.json`（模拟器写入位置）。  
- **顶层字段 `schemaVersion`（数字）**：表示状态 JSON 结构版本；当前模拟器写入 **1**。  
- 可选 **`config.loadedConfigPath`**：本次运行实际加载的配置绝对路径（便于核对「到底读了哪个文件」）。  
- 更正式的字段说明与校验：仓库内 [`schemas/sim-state-v1.schema.json`](../../schemas/sim-state-v1.schema.json)（JSON Schema，供工具或 CI 使用）。

---

## 6. 大节点数性能建议

- 设备数量越多，`sim-state.json` 体积越大，写盘与读取开销越高。若你不需要全量节点信息，建议在配置层面控制设备数或启用过滤/抽样策略（本页不展开实现）。

## 7. ChirpStack 设备与配置对齐

```bash

```

字段与 v2 `devices[]` / legacy `lorawan` 对齐说明见脚本头部注释及 [ChirpStack 联调配置核对清单](ChirpStack联调配置核对清单.md)。
