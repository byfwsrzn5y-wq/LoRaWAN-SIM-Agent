# 异常行为模板参考

本文为**参考文档**：列出各异常行为模板相对基线的偏离维度、与问题维度的对应关系及配置说明。行为机制与随机节点生成见 [行为模型与随机节点](行为模型与随机节点.md)，100 节点快速配置见 [100 节点配置指南](100节点正常与异常配置.md)。**节点完整功能正常行为基线**见 [行为模型与随机节点#一、节点完整功能正常行为基线](行为模型与随机节点.md#一节点完整功能正常行为基线)。

## 异常行为与基线偏离维度

各模板相对「基线」的偏离项（基线为完整功能正常行为；未列出的模板为仅距离/位置差异的正常变体）：

| 模板 ID | 相对基线的偏离维度 |
|---------|---------------------|
| normal-near / normal-mid / normal-far | 无偏离（仅 nodeState.rssi/snr 表示近/中/远） |
| bad-snr-1 / bad-snr-2 / bad-snr-3 | nodeState：SNR 差、抖动大（信号质量/覆盖） |
| rejoin-5s | 入网/会话保持：uplink.rejoinIntervalMs（频繁重入网） |
| good-sf7-adr-fail / sf12-good-signal / sf7-poor-link | ADR、SF：lorawan.dataRate 固定 + adrReject（ADR 不合理/SF 不合理） |
| high-rate-confirmed | 上行类型与间隔：confirmed + 高频率（流量/负载） |
| mac-heavy | 上行：linkCheckInterval、payloadLength、interval（MAC 为主、应用数据少） |
| low-battery | devStatus：battery/margin 异常（电量/裕量上报） |
| lossy / confirmed-lossy | 丢包/重传：uplinkDropRatio、confirmed（丢包率与重传策略） |
| join-then-sleep | 首包后间隔：intervalAfterFirstDataMs（入网后长睡） |
| duplicate-fcnt | 重复 FCnt：duplicateFirstData（重放/FCnt 处理） |
| burst-silence | 突发/静默：burstCount、burstIntervalMs、silenceAfterBurstMs（流量模式） |
| channel-single | 信道/频段：nodeState.channels 单信道（频段配置不合理） |
| link-jittery | nodeState：rssiJitter/snrJitter 大（链路波动/时间序列异常） |

## 一、与问题维度的模板对应（行为模型 ID）

| 应用场景/问题维度 | 数据来源(网关可观测) | 可用的行为模板 ID |
|------------------|----------------------|-------------------|
| 信号质量与覆盖优化 | RSSI、SNR、接收率 | normal-near/mid/far、bad-snr-1/2/3、link-jittery |
| 网络流量与负载优化 | 上下行流量、包数、节点数 | high-rate-confirmed、burst-silence、mac-heavy |
| 数据丢包率与重传策略 | 丢包率、confirmed 重传次数、时延 | lossy、confirmed-lossy、duplicate-fcnt |
| 网关状态与频段优化 | 频段使用、干扰信号、成功率 | channel-single（单信道/频段配置不合理） |
| **ADR 策略合理性(SF 分布)** | SF/DR、RSSI、SNR、丢包率 | good-sf7-adr-fail、**sf12-good-signal**、**sf7-poor-link** |
| LoRaWAN 帧类型诊断 | Join/Confirm/ACK/MAC 统计 | rejoin-5s、high-rate-confirmed、mac-heavy |
| 异常节点识别(行为/电量/配置) | 丢包、重传、ACK/Join 成功率、SF 分布 | low-battery、adr 相关模板、**channel-single**、**link-jittery** |
| 链路稳定性与时间序列异常 | 历史 RSSI/SNR、丢包、重传、Join/ACK | link-jittery、lossy、confirmed-lossy |

## 二、已支持的异常行为（含 ADR/SF/频段）

| 行为 | 配置方式 | 典型场景 |
|------|----------|----------|
| **电量/链路恶化上报** | 设备上设置 `devStatus: { battery: 0, margin: -20 }` | 电池即将耗尽、或接收裕量极差时，DevStatusReq 下行会得到低电量/负 margin 的 DevStatusAns，便于测试告警与策略 |
| **上行随机丢包** | `uplink.uplinkDropRatio: 0.2`（0~1） | 信道不稳定、设备移动导致的部分包丢失，ChirpStack 侧可见 FCnt 不连续 |
| **入网后发一包即长睡** | `uplink.intervalAfterFirstDataMs: 3600000` | 入网后只发一条数据即进入长时间休眠（如 1 小时），测试会话保持与 FCnt 续接 |
| **首包 FCnt 重复** | `uplink.duplicateFirstData: true` 或设备 `duplicateFirstData: true` | 设备异常或重发导致同一 FCnt 上行发两次，测试服务端重放与 FCnt 处理 |
| **突发后长静默** | `uplink.burstCount: 5`、`burstIntervalMs: 2000`、`silenceAfterBurstMs: 300000` | 先短时间连发数包（如 5 包/2s），再长时间静默（如 5 分钟），模拟按需上报型设备 |
| **ADR 不合理 - 高 SF 不必要** | 模板 `sf12-good-signal`：信号好却固定 DR0(SF12)、ADR 回 NACK | 空口占用大、SF 分布不合理，对应「ADR 策略合理性分析」中高 SF 低效节点 |
| **SF 不合理 - 弱链路用低 SF** | 模板 `sf7-poor-link`：弱链路(RSSI/SNR 差)却固定 DR5(SF7)、ADR 回 NACK | 易丢包、ADR/SF 与链路不匹配，对应「SF 不合理」 |
| **频段配置不合理** | 模板 `channel-single`：`nodeState.channels: [923.2]` 仅用单信道 | 单信道占用/自干扰或频点配置错误，对应「网关状态与频段优化」 |
| **链路波动大** | 模板 `link-jittery`：大 `rssiJitter`/`snrJitter` | 时间序列异常、链路不稳定，对应「链路稳定性趋势与时间序列异常」 |
| **确认包 + 高丢包（重传多）** | 模板 `confirmed-lossy`：`confirmed: true` + `uplinkDropRatio: 0.35` | 确认包重传多、负载与丢包率优化，对应「数据丢包率与重传策略」 |

## 三、配置说明

### 1. devStatus（DevStatusAns 内容）

- **battery**：0–255。0=外部供电或临界，1–254=电量等级，255=未知。
- **margin**：-32–31（6 bit 有符号）。接收裕量（dB），负值表示链路差。

示例：低电量 + 差裕量

```json
"devStatus": { "battery": 5, "margin": -20 }
```

### 2. uplinkDropRatio（上行丢包率）

- 0~1，每包以该概率**不发出**（不调 MQTT/UDP），但本节点仍按间隔继续调度。
- 用于模拟无线丢包，不改变 FCnt 递增逻辑（仅“少发”若干包）。

### 3. intervalAfterFirstDataMs（首包后间隔）

- 入网后**第一包 Data** 发出后，下一包发送间隔使用该值（毫秒），不再使用 `interval`。
- 适合“入网→发一包→长睡”的设备。

### 4. duplicateFirstData（首包 FCnt 重复一次）

- 为 `true` 时：入网后第一包 Data 发完后，下一包**复用同一 FCnt** 再发一次，再之后恢复正常递增。
- 用于测试服务端对重复 FCnt / 重放的处理。

### 5. burstCount + burstIntervalMs + silenceAfterBurstMs（突发后静默）

- **burstCount**：每轮连续发送包数。
- **burstIntervalMs**：突发内包间隔（毫秒）。
- **silenceAfterBurstMs**：每轮突发结束后的静默时长（毫秒）。

一轮内按 burstIntervalMs 发 burstCount 包，然后等待 silenceAfterBurstMs，再进入下一轮。

### 6. ADR/SF 与频段（对应图片中的 ADR 策略合理性、频段优化）

- **固定 DR（SF）**：在设备或模板中设置 `lorawan.dataRate`（0=SF12 … 5=SF7），并配合 `adrReject: true` 表示不接受 LinkADRReq，从而固定为「不合理」的 SF。
- **单信道**：在 `nodeState.channels` 中只配置一个频点（如 `[923.2]`），表示频段/信道配置不合理或单信道滥用。

## 四、其他常见真实场景（未单独实现或部分覆盖）

| 场景 | 说明 | 当前模拟器 |
|------|------|------------|
| 仅发 unconfirmed、从不开 RX | 设备只发 unconfirmed，不接收下行 | 已支持：配置为 unconfirmed 即可；RX 由服务端/网关侧体现 |
| 长时间静默后再次上线 | 数小时无上行后再次发数 | 可用 `interval` 很大或 `intervalAfterFirstDataMs` 模拟长睡 |
| FCnt 回绕 / 设备侧 FCnt 重置 | 设备重启导致 FCnt 从 0 开始 | 可用「重置」接口清会话或 FCnt 后重发 |
| 最大/最小 payload 长度 | 0 字节或 222 字节 | 支持 2–20 字节；0 或 222 需扩展 payload 生成逻辑 |
| 多 FPort | 使用 FPort 2、3 等 | 已支持在 `uplink.lorawan.fPort` 中配置 |
| 下行 ACK 丢失（confirmed 上行未收到 ACK） | 设备会重发 | 由 ChirpStack/网关侧处理；模拟器只发 confirmed |

## 五、异常节点配置方式

推荐使用**行为模板 + 随机节点**：在 `configs/behavior-templates.json` 中已有对应模板（如 `low-battery`、`lossy`、`join-then-sleep` 等），通过 `lorawan.randomBehaviors` + `behaviorTemplateList` 或 `behaviorCounts`/`behaviorRatios` 即可控制种类与数量，见 [行为模型与随机节点](行为模型与随机节点.md)。

若使用 **config.devices** 手动列举设备，可参考以下片段：

```json
{
  "name": "node-11-low-battery",
  "nodeState": { "rssi": -80, "snr": 5 },
  "devStatus": { "battery": 10, "margin": -15 },
  "uplink": { "interval": 30000 }
},
{
  "name": "node-12-lossy",
  "nodeState": { "rssi": -88, "snr": 3 },
  "uplink": { "interval": 20000, "uplinkDropRatio": 0.25 }
},
{
  "name": "node-13-join-then-sleep",
  "nodeState": { "rssi": -75, "snr": 6 },
  "uplink": { "interval": 10000, "intervalAfterFirstDataMs": 3600000 }
},
{
  "name": "node-14-duplicate-fcnt",
  "nodeState": { "rssi": -78, "snr": 5 },
  "uplink": { "interval": 30000, "duplicateFirstData": true }
},
{
  "name": "node-15-burst-silence",
  "nodeState": { "rssi": -82, "snr": 4 },
  "uplink": { "burstCount": 5, "burstIntervalMs": 2000, "silenceAfterBurstMs": 300000 }
}
```

以上能力均可在现有配置与代码下使用，用于贴近真实场景的压力与异常测试。
