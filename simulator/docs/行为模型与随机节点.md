# 行为模型与随机节点

说明**行为基线**、**行为模板**（baseline/extends）与**随机节点生成**（randomBehaviors、behaviorTemplateList）。100 节点快速配置见 [100 节点配置指南](100节点正常与异常配置.md)，异常模板与问题维度参考见 [异常行为模板参考](异常行为模板参考.md)。

先在一处定义**节点完整功能正常**的基线行为，再通过**行为模板**描述正常变体（如远近）与异常（相对基线的偏离）；随机生成节点时从模板池中匹配，后续可在此池中持续新增模板。

## 一、节点完整功能正常行为基线

以下为「节点完整功能正常」时各维度的**正常取值**，与 `configs/behavior-templates.json` 中的 `baseline` 及实现一致。异常行为即在这些维度上的偏离。

| 维度 | 正常取值 | 说明 |
|------|----------|------|
| 入网 | OTAA，Join 一次后保持会话 | 不频繁重入网 |
| 上行间隔 | 30000 ms（可配置） | 基线取该默认值 |
| 上行类型 | Unconfirmed | 非高负载 confirmed 风暴 |
| ADR | 接受 LinkADRReq（不设 adrReject） | SF/功率随网络侧调整 |
| 信道 | 区域默认信道列表（如 AS923-1 多信道） | 非单信道 |
| nodeState | RSSI/SNR 在合理范围，抖动适中（如 rssiJitter 1.5、snrJitter 0.8） | 链路稳定 |
| devStatus | 不报低电量/极差裕量 | 可选上报，基线不设异常值 |
| 丢包/重传 | 无模拟丢包（uplinkDropRatio 0）、无重复 FCnt | 链路稳定 |
| 首包后间隔 | 与常规间隔一致 | 无「发一包即长睡」 |
| 突发/静默 | 无 burst，常规间隔 | 无异常突发模式 |
| MAC | 可选 LinkCheck 等，应用数据为主 | 非「仅 MAC 几乎无应用数据」 |

各异常模板相对基线的**偏离维度**对照见 [异常行为模板参考#异常行为与基线偏离维度](异常行为模板参考.md#异常行为与基线偏离维度)。

## 二、模板定义（baseline + extends）

- **位置**：`configs/behavior-templates.json`（或通过 `lorawan.behaviorTemplatesFile` 指定）。
- **baseline**：顶层 `baseline` 对象描述「完整功能正常」的默认行为（nodeState、uplink 等）。所有带 `extends: "baseline"` 的模板先继承 baseline，再仅写**差异项**。
- **结构**：`templates` 为对象，key 为模板 ID，value 为偏置配置；可选 `extends: "baseline"` 表示相对基线只写偏离项。

示例（带 baseline 与 extends）：

```json
{
  "baseline": {
    "description": "节点完整功能正常行为",
    "nodeState": { "rssi": -80, "snr": 5, "rssiJitter": 1.5, "snrJitter": 0.8 },
    "uplink": { "interval": 30000 }
  },
  "templates": {
    "normal-near": {
      "extends": "baseline",
      "description": "正常-近端：信号好",
      "nodeState": { "rssi": -72, "snr": 7 }
    },
    "rejoin-5s": {
      "extends": "baseline",
      "description": "频繁重入网-每5s重置",
      "uplink": { "interval": 5000, "rejoinIntervalMs": 5000 }
    }
  }
}
```

- **可选字段**（与 `config.devices[].*` 一致）：
  - `description`：说明，仅文档用，不参与合并。
  - **extends**：设为 `"baseline"` 时，先应用 baseline 再合并本模板字段（仅写相对基线的差异即可）。
  - `nodeState`：rssi、snr、rssiJitter、snrJitter、**channels**（如单信道 `[923.2]` 表频段配置不合理）等。
  - `uplink`：interval、payloadLength、linkCheckInterval、confirmed、uplinkDropRatio、intervalAfterFirstDataMs、duplicateFirstData、burstCount、burstIntervalMs、silenceAfterBurstMs、rejoinIntervalMs 等。
  - `lorawan`：如 **dataRate**（0=SF12 … 5=SF7，用于 ADR/SF 不合理场景）。
  - `devStatus`：battery、margin（DevStatusAns）。
  - **adrReject**：boolean，对 LinkADRReq 回 NACK（可模拟 ADR 不合理、SF 固定）。
  - `duplicateFirstData`：boolean 或置于 uplink 中，首包 FCnt 重复一次。

未写 `extends: "baseline"` 的模板保持原有语义：仅用该模板自身字段（与旧版兼容）。新增异常模板时建议使用 `extends: "baseline"` 并只写相对基线的偏离项。模板池与问题维度的对应关系见 [异常行为模板参考](异常行为模板参考.md)。

## 三、随机节点生成

在 `lorawan` 中开启随机行为并指定设备数与模板来源：

```json
"lorawan": {
  "enabled": true,
  "activation": "OTAA",
  "appEuiStart": "0000000000000001",
  "devEuiStart": "0102030405060701",
  "appKey": "00112233445566778899AABBCCDDEEFF",
  "randomBehaviors": true,
  "deviceCount": 20,
  "behaviorTemplatesFile": "configs/behavior-templates.json",
  "behaviorTemplateList": "all"
}
```

| 字段 | 说明 |
|------|------|
| `randomBehaviors` | 为 `true` 时，按 `deviceCount` 生成 OTAA 设备，每个设备随机选用一条模板（不再使用 `config.devices` / CSV）。 |
| `deviceCount` | 生成的节点数量。 |
| `behaviorTemplatesFile` | 模板 JSON 路径（相对 cwd 或绝对）。也可用内联 `behaviorTemplates: { "id": { ... } }`。 |
| `behaviorTemplateList` | 参与随机的模板 ID 列表。`"all"` 或省略：使用文件/内联中的全部模板；**数组**如 `["normal-near","bad-snr-1","lossy"]`：仅从这些 ID 中随机，即**任意指定多少种不同行为**（正常+异常）。 |

- **随机 100 个节点、指定 10 种行为**：设 `deviceCount: 100`，`behaviorTemplateList` 为长度为 10 的数组（如 3 正常 + 7 异常），则 100 个节点每个随机命中这 10 种之一；种类数由数组长度决定。示例配置见 `configs/config-100nodes-10types.json`。
- 设备名：`node-{序号}-{模板ID}`，例如 `node-1-normal-near`、`node-2-rejoin-5s`。
- 启动后控制台会打印「按行为模板随机生成 N 个 OTAA 设备」及**模板分布**（各模板被选中的次数）。

## 四、与其它配置的优先级

- 若配置了 **CSV**（`lorawan.csvImportPath`），优先从 CSV 加载，不会走随机模板。
- 若配置了 **config.devices**（数组且含 `lorawan`/`activation`），优先从 `config.devices` 加载，不会走随机模板。
- 仅当**未**使用 CSV 且**未**使用 `config.devices`，且 `lorawan.randomBehaviors === true` 且 `deviceCount > 0` 时，才从行为模板池中随机匹配生成节点。

## 五、新增行为模板

1. 打开 `configs/behavior-templates.json`（或你配置的 `behaviorTemplatesFile`）。
2. 在 `templates` 中增加一条 key（模板 ID）和 value。建议使用 `extends: "baseline"` 并只写**相对基线的偏离项**：

```json
"my-new-behavior": {
  "extends": "baseline",
  "description": "自定义：例如每小时发一包",
  "uplink": { "interval": 3600000 }
}
```

3. 若使用 `behaviorTemplateList: "all"`，新模板会自动参与随机；若为数组，需把 `"my-new-behavior"` 加入该数组。

无需改代码即可持续扩展行为类型；先定清基线、再写差异，便于维护。

## 六、示例配置

- **随机 20 节点、全部模板**：`configs/config-random-behaviors.json`（`deviceCount: 20`，`behaviorTemplateList: "all"`）。
- **随机 100 节点、指定 10 种行为**：`configs/config-100nodes-10types.json`（`deviceCount: 100`，`behaviorTemplateList` 为 10 个模板 ID 的数组）；可任意修改节点总数和数组内容以指定「多少种不同的异常/正常」。
- **仅用部分模板**：在 `lorawan` 中设置 `"behaviorTemplateList": ["normal-near", "normal-mid", "lossy", "low-battery"]`，节点数由 `deviceCount` 决定。

ChirpStack 侧需创建与 `deviceCount` 相同数量的 OTAA 设备，DevEUI 按 `devEuiStart` 递增（如 20 个：`0102030405060701` … `0102030405060714`），AppKey 与配置一致。
